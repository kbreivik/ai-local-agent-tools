End-to-end traffic flow from browser to infrastructure and back. Each step is a distinct processing stage.

1
Browser → Login Request
User submits credentials in the Vue SPA (served from FastAPI :8000 as static files).
POST /api/auth/login {username, password}
No token required — this is the public endpoint.
CORS allowed from all origins (CORS_ALLOW_ALL=true).
2
FastAPI — JWT Issuance
api/auth.py verifies bcrypt hash (ADMIN_PASSWORD env). On success issues a 24-hour JWT signed with JWT_SECRET.
bcrypt.verify(password, stored_hash)
jwt.encode({sub: username, exp: now+24h}, JWT_SECRET, HS256)
← 200 {token: "eyJ..."}
Browser stores token in localStorage → used in all subsequent requests.
3
Browser → Task Submission
User types a task in the GUI and clicks Run. The GUI sends the task text and a session_id.
POST /api/agent/run
Headers: Authorization: Bearer <jwt>
Body: {task: "check kafka consumer lag", session_id: "..."}

Simultaneously browser opens WebSocket for live output:
WS /ws/output?token=<jwt>
4
FastAPI — Auth + Lock Guard
Every protected route verifies the JWT. Destructive operations additionally check the PlanLockManager — only one pending plan_action is allowed across all sessions at once.
get_current_user() → jwt.decode(token, JWT_SECRET)
PlanLockManager.is_locked() → if locked, return {status:"locked"}
Agent loop starts in background thread.
5
Agent Loop — Task Routing
classify_task() assigns the task to one of three agents: ORCHESTRATOR (planning, multi-step), RESEARCH (read-only, info gathering), or EXECUTION (single destructive action). Each agent has a filtered tool allowlist.
classify_task(task) → "orchestrator" | "research" | "execution"
filter_tools(agent_type) → allowed tool names list
get_prompt(agent_type, memory_context) → system prompt

Memory activation: POST /api/memory/activate → MuninnDB :9475
Returns relevant past engrams → injected into system prompt.
6
LLM Inference — Tool Selection
OpenAI SDK sends the system prompt, task, and tool schemas to LM Studio. The LLM responds with a tool_call JSON or a plain text answer.
POST http://192.168.199.51:1234/v1/chat/completions
model: lmstudio-community/qwen3-coder-30b-a3b-instruct
tools: [...allowed MCP tool schemas...]

Response: {choices: [{finish_reason: "tool_calls", message: {tool_calls: [...]}}]}
— or —
Response: {choices: [{finish_reason: "stop", message: {content: "..."}}]}
7
Agent Loop — Plan Guard Check
If the tool is plan_action (destructive operation), the agent pauses and broadcasts a confirmation request to the WebSocket. Execution blocks until the user approves or cancels via POST /api/agent/confirm.
tool_call.name == "plan_action"?
→ PlanLockManager.acquire(session_id)
→ WS broadcast: {type:"confirm_request", plan: {...}}
→ WAIT for /api/agent/confirm {approved: true/false}

Non-destructive tools: execute immediately, no confirmation.
8
Tool Execution — External Services
tool_registry.invoke_tool() calls the appropriate MCP tool function. Tools reach out to external services as needed.
Examples:
kafka_consumer_lag() → kafka1:9092 (kafka-python)
elastic_search_logs() → ELASTIC_URL:9200 (requests)
proxmox_vm_status() → PROXMOX_HOST:443 (proxmoxer)
docker_swarm_state() → /var/run/docker.sock (docker SDK)
skill_execute(name) → runs skill module execute(**kwargs)
muninn_store(engram) → localhost:9475 (requests)

All tools return: {status:"ok"|"error"|"degraded", data:..., timestamp:..., message:...}
9
LLM — Result Synthesis
Tool result appended to messages as a tool_result. LLM called again to synthesise a human-readable response. Loop continues until finish_reason="stop" or max iterations reached.
messages.append({role:"tool", tool_call_id: id, content: json(result)})
POST /v1/chat/completions ← second (or Nth) LLM call
finish_reason == "stop"? → final text answer ready
finish_reason == "tool_calls"? → loop back to step 6
10
WebSocket Broadcast → Browser
Each agent step (tool call, tool result, final answer) is broadcast in real time over the WebSocket. Lines are also persisted to operation_log for replay on reconnect.
WS message types:
tool_call {name, args} — LLM chose a tool
tool_result {name, result} — tool returned data
confirm_request {plan} — awaiting human approval
agent_response {text} — final answer text
error {message} — runtime error

All lines → operation_log table (async 200ms flush)
On WS reconnect: replay lines from DB for session continuity.
11
Memory Storage — Post-Session
After session completes, significant outcomes stored as engrams in MuninnDB for future context retrieval.
POST localhost:9475/api/engrams
{content: summary, tags: [service, operation, outcome]}

Next session: activate_memory(new_task) fetches relevant engrams
→ injected into LLM system prompt at step 5.
