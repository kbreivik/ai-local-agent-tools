# Skill Lifecycle & Focused Agent Architecture Design

## Goal

Three tightly related improvements:
1. **Skill persistence** — generated skill files survive container rebuilds
2. **Skill lifecycle GUI** — test → promote to `@mcp.tool()` or scrap, with domain assignment
3. **Focused agent routing** — intent+domain two-level routing, step-through orchestrator with structured verdicts and service-aware gates

---

## Architecture

### Skill Persistence

The loader scans **two directories** on startup:

| Path | Purpose | Persisted? |
|------|---------|-----------|
| `mcp_server/tools/skills/modules/` | Image-baked starter skills | Always (in image) |
| `data/skill_modules/` | Generated + promoted skills | Yes (agent-data volume) |

Generated skills write to `data/skill_modules/`. No new Docker volume needed — `agent-data` already covers `/app/data/`. Starter skills (proxmox_vm_status, fortigate_system_status, http_health_check) remain in the image and are always available.

### Skill Lifecycle States

```
auto_generated → [tested] → promoted
                          ↘ scrapped → [restore] → auto_generated
promoted → [demoted] → auto_generated
```

**Transitions:**
- `auto_generated` → `promoted`: user tests skill, result is acceptable, clicks Promote, picks domain
- `auto_generated` → `scrapped`: user tests skill, result is wrong, clicks Scrap
- `promoted` → `auto_generated`: user clicks Demote (removes @mcp.tool() wrapper on next restart)
- `scrapped` → `auto_generated`: user clicks Restore (re-enables skill file)

**DB changes:** Add `lifecycle_state` column (`auto_generated` | `promoted` | `scrapped`) and `agent_domain` column (`proxmox` | `kafka` | `swarm` | `general` | `build` | null).

### Skill Promotion Mechanics

When a skill is promoted:
1. Backend writes a `@mcp.tool()` wrapper to `mcp_server/tools/promoted_skills.py` (appended, auto-generated, do not edit manually)
2. DB marks skill `lifecycle_state="promoted"`, `agent_domain=<chosen>`
3. `router.py` domain allowlist for that agent is extended with the skill name at runtime (loaded from DB on startup)
4. Takes effect on next process restart. GUI shows badge "promoted · active on restart"

`promoted_skills.py` is **regenerated from scratch** on every promote or demote operation — the backend reads all promoted skills from DB and rewrites the entire file. Never append/delete individual lines. This ensures the file stays consistent with DB state across restarts.

`promoted_skills.py` format:
```python
# AUTO-GENERATED by skill promotion system — do not edit manually
# Each wrapper calls skill_execute() under the hood
from mcp_server.tools.skills.loader import dispatch_skill

@mcp.tool()
def proxmox_vm_monitor(node: str = "", vmid: str = "") -> dict:
    """Monitor Proxmox VM CPU and memory utilisation via API."""
    return dispatch_skill("proxmox_vm_monitor", node=node, vmid=vmid)
```

### Focused Agent Routing

Two-level routing replaces the current single `classify_task()`:

**Level 1 — Intent:**
| Intent | Replaces | Tools |
|--------|---------|-------|
| `observe` | `status` | All read-only tools, any service (~15) |
| `investigate` | `research` | Logs, correlation, MuninnDB search (~12) |
| `execute` | `action` | Destructive infra tools, filtered by domain (~6–10) — skill management tools removed |
| `build` | _(new)_ | Skill create/test/promote/manage (~8) — takes over all skill_* tools from action agent |

Skill management tools (`skill_create`, `skill_regenerate`, `skill_disable`, `skill_enable`, `skill_import`) move exclusively to `build`. The `execute` agent no longer includes them. Tasks mentioning "skill", "create skill", "generate skill" route to `build`.

**Level 2 — Domain (execute only):**
| Domain | Tools |
|--------|-------|
| `kafka` | pre_kafka_check, kafka_broker_status, kafka_rolling_restart_safe, kafka_topic_health, kafka_consumer_lag, plan_action |
| `swarm` | swarm_status, service_list, service_health, service_upgrade, service_rollback, node_drain, pre_upgrade_check, post_upgrade_verify, plan_action |
| `proxmox` | Promoted proxmox skills + plan_action |
| `general` | All non-destructive tools + plan_action |

Domain detection: keyword scan of task text (`"kafka"`, `"broker"`, `"topic"` → kafka; `"swarm"`, `"service"`, `"stack"` → swarm; `"proxmox"`, `"vm"`, `"node"` → proxmox).

### plan_action Lock and the Orchestrator

Each `execute` step acquires `plan_action` individually (existing behavior unchanged). The orchestrator does **not** acquire a global lock for the whole plan upfront. This means:
- Step 1 (observe) runs freely — no lock
- Step 2 (execute) calls `plan_action()` as usual — lock acquired, user approves, lock released
- If the user has another session holding the lock when step 2 runs, step 2 blocks and shows "another operation in progress"

This keeps the existing session isolation guarantees intact.

### Step-Through Orchestrator

For multi-domain tasks the orchestrator builds a **step plan** and executes steps sequentially.

**Step plan example** — "check swarm health then rolling restart kafka brokers":
```
Step 1: observe   [all domains]  → check pre-conditions
Step 2: execute   [kafka]        → rolling restart
```

**Context between steps** is a minimal structured verdict (~50 tokens), not prose:
```json
{"step": 1, "verdict": "GO", "facts": {"brokers": "3/3", "quorum": true, "lag": 0}}
```

**Verdict values:**
- `GO` — all gate rules pass, proceed automatically
- `ASK` — knowledge gap (e.g. no changelog ingested for target version) — present options to user
- `HALT` — gate rule failure that would cause outage — block execution, explain why

**ASK options** (presented in output panel):
1. Ingest missing knowledge and re-run this step
2. Skip check and proceed (user accepts risk)
3. Cancel

### Service-Aware Gate Rules

Stored in `api/agents/gate_rules.py`. Each rule is a pure function: `(facts: dict) -> Verdict`.

| Service | Rule | HALT condition |
|---------|------|----------------|
| kafka rolling restart | All brokers up, ISR ≥ RF−1 | Any broker offline, or restart would drop below quorum |
| swarm service upgrade | Manager quorum ≥ ⌈n/2⌉+1, replica count ≥ 2 | Would leave cluster without quorum |
| proxmox VM restart | If HA member: check HA quorum | HA quorum would be lost |
| proxmox VM restart | If standalone: always safe | — |
| version upgrade (any) | Changelog ingested for target version | No changelog → ASK |

## API Endpoints

```
POST   /api/skills/{name}/promote    body: {"domain": "kafka|swarm|proxmox|general"}
POST   /api/skills/{name}/demote     (no body)
DELETE /api/skills/{name}            scrap — disables, removes file, keeps DB record
POST   /api/skills/{name}/restore    re-enables scrapped skill
```

## GUI Changes

### SkillsPanel — skill card states

**Untested** (`auto_generated`, no test run yet):
- Promote and Scrap buttons disabled
- Tooltip: "Run a test first to unlock"

**Tested** (test has been run at least once):
- Promote button enabled (green)
- Scrap button enabled (red)

**Promoted** (`lifecycle_state="promoted"`):
- Green border, "promoted" badge with domain tag
- Shows "active on restart" if promoted since last start
- Demote button replaces Promote

**Scrapped** (`lifecycle_state="scrapped"`):
- Greyed out, strikethrough name, red "scrapped" badge
- Restore button only

### Promote dialog
Inline in the card (not a modal). Shows domain options:
- proxmox / swarm / kafka / general
- Domain pre-selected based on skill category from SKILL_META
- Confirm button writes promotion

### Output panel — step execution
Each step renders as a verdict card:
- Header: step number, intent, domain, verdict badge (GO/ASK/HALT)
- Body: structured check results (key → value pairs)
- HALT: explanation of which rule failed and what would be lost
- ASK: three action buttons (ingest / skip / cancel)
- GO: auto-proceeds to next step after brief display
