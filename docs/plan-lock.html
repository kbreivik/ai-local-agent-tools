Plan Lock & Destructive Operation Safety

How HP1-AI-Agent prevents conflicting changes across concurrent sessions
Overview

A global PlanLockManager singleton enforces that only one destructive operation can be pending approval at a time — across all sessions and all users. Read-only operations are never affected. The lock is acquired when the LLM calls plan_action(), held until the human approves or cancels in the GUI, then released automatically.

Read-only ops: always pass through plan_action(): acquires lock → confirmation required Destructive tools without plan_action: blocked by pre-flight guard Lock held by another session: destructive ops blocked
Execution Flow — Single Session
1
LLM decides an action is needed
Agent loop receives a task. LLM determines a destructive operation is required (e.g. restart a Kafka broker, upgrade a Swarm service, create a skill).
Task: "restart kafka broker 2 — consumer lag is critical"
LLM: finish_reason = "tool_calls" → wants to call kafka_rolling_restart_safe
2
Pre-flight guard — plan_action must come first
Before any destructive tool executes, the agent loop checks whether plan_action() has already been called in this turn. If not, every destructive tool call in the batch is blocked and a mandatory reminder is injected into the LLM conversation.
BLOCKED kafka_rolling_restart_safe — plan_action() not yet called
Synthetic tool result: "plan_action() must be called before this tool."
Injected user message: "MANDATORY: call plan_action() as a TOOL first."
→ LLM re-prompted; loop continues
3
LLM calls plan_action() — global lock acquired
LLM calls plan_action() with a structured plan. The agent tries to acquire the PlanLockManager singleton. Two outcomes:
✓ Lock free → acquired
Lock granted to this session_id + owner_user.
GUI receives confirm_request WS message.
Agent loop suspends — waiting for human.
✗ Lock held by other session
Returns {status:"locked", approved:false}.
LLM is told to wait — operation cannot proceed.
No confirmation modal shown.
plan_action({
  summary: "Rolling restart kafka broker 2",
  steps: ["drain consumer group", "restart broker", "verify ISR"],
  risk_level: "high",
  reversible: false
})
→ PlanLockManager.acquire(session_id, owner_user) → True
→ WS broadcast: {type: "confirm_request", plan: {...}}
4
Human reviews & decides in GUI
The confirmation modal appears in the browser. The plan summary, steps, risk level, and reversibility are shown. No timeout — lock is held until explicitly resolved.
✓ User clicks Approve
POST /api/agent/confirm {approved: true}
Agent loop resumes → destructive tool executes.
✗ User clicks Cancel
POST /api/agent/confirm {approved: false}
Agent loop logs cancellation → stops.
Lock released immediately.
5
Destructive tool executes
With approval confirmed, the agent loop calls the actual destructive tool. Output streamed over WebSocket in real time.
invoke_tool("kafka_rolling_restart_safe", {broker_id: "2"})
→ SSH to broker host, drain, restart, wait for ISR sync
→ WS: {type:"tool_result", name:"kafka_rolling_restart_safe", result:{status:"ok",...}}
Lock still held during execution — prevents concurrent destructive ops
6
Lock released — session ends
Lock is released when: tool completes, session errors out, or user force-releases via POST /api/lock/force-release. Any queued session can now acquire the lock.
plan_lock.release(session_id)
Lock free — other sessions unblocked

Lock Behaviour Rules
Scenario Behaviour
Destructive tool called without plan_action() first Blocked — synthetic error result injected, LLM re-prompted with mandatory instruction to call plan_action() first. Loop continues.
Text response to a destructive task (LLM forgets to call tools) Intercepted — if task contains destructive keywords and no plan_action() was called, a forced reminder is injected and the loop continues (up to max_steps − 2).
Lock free — first session calls plan_action() Lock acquired — GUI confirmation modal shown. Agent suspends.
Lock held — second session calls plan_action() Lock denied — returns {status:"locked", approved:false}. LLM told to wait. No modal shown.
Same session calls plan_action() again (re-entrant) Re-granted — same session_id already holds the lock; acquires without blocking.
User approves in GUI Execution proceeds — lock remains held during tool execution, released on completion.
User cancels in GUI Cancelled — lock released immediately. Session logs cancellation and stops.
Session errors or crashes Auto-release — session cleanup calls plan_lock.release(session_id) in the finally block. Lock freed even on exception.
Lock stuck (session crashed without cleanup) Force-release via POST /api/lock/force-release. Requires authentication. Available to any logged-in user.
Read-only tool — any session, any time Always allowed — lock is never consulted for non-destructive tools.
Implementation — Key Files
File Responsibility
api/lock.py PlanLockManager singleton — acquire, release, force_release, is_locked_by_other. asyncio.Lock for thread safety.
api/routers/agent.py — DESTRUCTIVE_TOOLS frozenset of tool names that require plan_action(). Checked in pre-flight guard and inter-session block.
api/routers/agent.py — _stream_agent() Pre-flight guard (blocks destructive without plan_action), inter-session lock check, plan_action intercept logic, confirmation wait loop, finally block auto-release.
api/routers/lock.py GET /api/lock/status — current lock state. POST /api/lock/force-release — admin override.
gui/src/components/LockBadge.jsx Orange 🔒 badge shown in GUI header when global lock is held. Polls /api/lock/status.
The lock is in-memory only — it does not survive a server restart. If the agent container restarts while a lock is held, the lock is automatically gone. For the same reason, force-release is only needed when a session crashes without triggering the finally block (rare).
